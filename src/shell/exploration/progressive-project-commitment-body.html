Progressive Project
Commitment — proposed flow

This page stands alone as a proposal for ASC ConnectED: start with no project, build a temporary working set, require a project only when the workflow needs a container, hand selections into that project, then work across modules—or step back to open exploration. Product and engineering can adopt, adapt, or challenge any step independently.

PHASE 01

Free exploration

Browse and compare without a project; optional save to project only when the user chooses.
01

Entry — no project selected

Explore
What's happening

The user lands without a project selected. This flow treats that as a first-class state: greeting, Catalog and BOM as quick entry, and recent projects—not an empty or “broken” shell.

Why it matters

The dashed project picker signals optional context, not an error. Modules that are safe to open without a project should stay visible so exploration does not feel blocked.

02

Explore catalog freely

Explore
What's happening

Full catalog access without a project. Search, filters, and "+ Compare" are all available. No upsells, no project-creation prompts.

Why it matters

This is where trust gets earned. A user who can browse without commitment will commit later when it's actually useful. Forcing a project here is the fastest way to lose first-time users.

03

Build a working set — soft engagement

Soft
What's happening

The user keeps a temporary comparison while still in explore mode. Save to project is visible but not required—soft engagement only.

Why it matters

Letting people assemble a set before naming a container earns intent. The hard gate comes only when a module needs a project (e.g. BOM), not here.

PHASE 02

Commit & attach

Hard gate → project-required modal (select / create / cancel) → pick or create → brief handoff while context transfers.
04

Commit action — hard gate (e.g. BOM)

Trigger
What's happening

The user clicked into BOM (or tried to "Add to BOM" from a product card). The empty state explains why a project is needed before introducing the prompt.

Why it matters

The framing is "a BOM needs a home" — not "you need to create a project to continue." Constraints feel helpful when they explain the underlying logic instead of stating a rule.

05

Project required

Trigger
Modal pattern

Three actions: Select a project, Create new project, or Cancel. Copy centers the work (“this needs a project”), not the user as the problem.

Implementation note

If engineering adds alternate escape paths (e.g. drafts), document them as extensions to this proposal—not silent behavior changes.

06

Select or create a project

Project
What's happening

After Select a project from the modal, this flow shows a searchable recent projects list plus a clear path to create new.

Why it matters

Separating this from the empty BOM keeps the commit moment legible: first acknowledge the rule, then pick the container.

07

Minimal project creation

Project
What's happening

One required field. Type is optional with a sensible default. Members, permissions, and settings are deferred to after creation.

Why it matters

The friction has to be proportional to the moment. The user just clicked "create" — they haven't earned a configuration form. Configuration belongs inside the project, not in the doorway.

08

Context transfer

Project
Handoff moment

An explicit loading beat makes the handoff feel intentional, not like a silent jump.

Product tie-in

Use the same screen for real async work (API attach) or a short timed state in prototypes—keep the project name visible.

PHASE 03

Inside the project

Active project chrome, then how work carries across modules (steps 9–10).
09

Project active — selections in context

Project
What's happening

The 3 items the user had selected pre-project now live in the BOM. The success banner makes the transfer visible and dismissible.

Why it matters

If the selection didn't carry over, the whole gate would feel like punishment for committing. Continuity of work is the contract the prompt makes with the user.

10

Cross-module persistence

Project
What's happening

A simple BOM → Submittals → Approvals read shows why the project container matters beyond a single screen.

Build implication

Engineering should keep one project scope in client and API so these modules read/write the same ids—not three disconnected silos.

PHASE 04

Mobility

Switch projects or step back to explore without losing your place in the app (steps 11–12).
11

Project switcher

Project
What's happening

From the project name in the top bar, the user opens recent projects and create new—mobility without leaving the shell.

Why it matters

Switching has to feel safe; otherwise users avoid committing. Keep context and route stable when only projectId changes.

12

Back to explore

Explore
Closure

Step 12 makes no-project an explicit return state, not a dead end—aligned with steps 1–2.

Product note

Preserve any in-memory working set policy in spec (clear vs keep) when exiting project context; this proposal assumes users can browse freely again.

Principles for this proposal

  1. No-project is a first-class state. Entry and return (steps 1 and 12) use the same dashed picker and explore affordances—not an error or empty void.
  2. Commit only when the workflow requires it. Catalog-style exploration stays open; the hard gate appears at project-bound actions (e.g. BOM) with an explicit in-page cue.
  3. Commit modal pattern: Select a project, Create new project, or Cancel—then list/create flows as shown. Extensions (e.g. drafts) should be documented as deliberate add-ons.
  4. Continuity of selection is the contract. Pre-project selections must land in the chosen project; the handoff step (8) makes that promise visible.
  5. One project context across modules. BOM, Submittals, and Approvals read the same active project so work does not fragment.
  6. Mobility without trap doors. The switcher supports changing projects or creating new ones; users can step back to explore when they choose.
  7. Minimal creation at the door. Name (and light metadata such as type) only; members, settings, and templates come after the project exists.
  8. Language carries the constraint. Prefer “this needs a project” and explain where BOMs live—avoid blaming the user.
